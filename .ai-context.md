# AI Context Map (.ai-context)

This file acts as the internal map for AI agents working on **Flappy Drone**. Read this file first to understand the structure, dependencies, and rules of the project.

## 1. Architecture Map
The project is a 2D physics game built in **Godot 4.3+**, currently supporting two isolated game modes:

- **Squash Mode** (`scenes/squash.tscn`, `scripts/squash.gd`)
  - **Purpose:** A local 1v1 / singleplayer pong-like arena.
  - **Key Logic:** Handles ball spawn, bounce decay, screen-wrapping via `WorldWrapper`, and custom rules defined in `squash_rules.gd`.
  
- **Sidescroller Mode** (`scenes/main.tscn`, `scripts/main.gd`)
  - **Purpose:** An infinite-scrolling obstacle avoidance game.
  - **Key Logic:** Coordinates the moving camera, spawns obstacles, tracks gates for score, and triggers Game Over on left-edge hit.

- **Player Controller** (`scenes/copter.tscn`, `scripts/copter.gd`)
  - **Purpose:** The core `RigidBody2D` player entity.
  - **Key Logic:** Exposes thrust, rotation, and dash mechanics based on configurable `PlayerConfigResource`. Fully isolated from game-mode logic.

## 2. Dependency Graph & Data Flow
How data flows between the major components:

- **Inputs -> Copter.gd:** The player scene takes raw Godot Input Actions (`ui_up`, etc.) and converts them to physics forces (Torque/Central Force).
- **ProceduralManager -> Main.gd:** In the sidescroller, `procedural_manager.gd` instantiates obstacles and `Area2D` score gates. When a `Copter` touches a gate, the manager emits `score_gained` to `main.gd`.
- **CameraSystem -> Main.gd:** The camera moves automatically. If a `Copter` falls behind and hits the `LeftEdgeDetector` Area2D, `CameraSystem` emits `player_fell_behind`, which `main.gd` catches to trigger a Game Over.
- **WorldWrapper -> Squash.gd**: In squash mode, objects leaving the screen are captured by `world_wrapper.gd`. It emits a `wrapped(axis, edge)` signal, which `squash.gd` uses to award points or reset the rally.

## 3. Coding Standards & Patterns
- **Godot 4.x Syntax:** Always use GDScript 2.0. Use typed variables (e.g., `var score: int = 0`) and strict return types (`-> void`).
- **Signal-Driven Architecture:** Avoid tight coupling. Children/Managers should `emit()` signals, and the parent (`main.gd` or `squash.gd`) should connect to those signals and coordinate the game state. Never `get_parent()` to call a method up the tree.
- **Atomic Components:** Keep new features small. E.g., The `CameraSystem` handles its own movement and edge detection independently.
- **Multiplayer Ready:** Write player logic to be agnostic of "Player 1 vs Player 2". Rely on injected configs (`player_config.gd`).

## 4. Ignore Rules & Context Diet
To keep the AI's active memory optimized, the following rules apply:
- **Ignored Directories:** `.git/`, `.godot/`, `.import/`, and `*.gd.uid` / `*.import` files are globally ignored in `.cursorignore`. Do not attempt to read or index them.
- **Context Chunking:** When solving a bug, do NOT read the entire `squash.gd` or `main.gd` if it pertains to physics. Isolate the read to `copter.gd` and related resources.
- **Future Refactor Target (Phase 3):** Local multiplayer logic is currently duplicated in modes. Before implementing online multiplayer, this will be extracted into a standalone `MultiplayerManager`.
